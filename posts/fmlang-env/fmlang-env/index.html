A toy RL environment for expression generation

29.12.2022 12:33 Reinforcement Learning

In Carper AI’s OpenELM project, we use diff models (trained on GitHub commits) to generate/mutate codes. The models are transformer-based and they are modelling codes as sequences. The rough idea is that the sequence models on codes will get feedbacks on its execution results, and Reinforcement Learning in turn uses the rewards to finetune and align the sequence modelling to the task. This is almost a partial analog of RLHF pipeline with a fixed and deterministic offline reward function (in a sense, feedback from the compiler).

There are some questions that one can ask deeper. One of the most obvious questions is

Why do we model codes using sequence modelling? Shouldn’t they be more naturally regarded as trees?

There has been a long history of modelling codes as trees. One can easily find a lot of treatments in the Genetic Programming literature. The recent interest in code generations as sequence models is perhaps from the tailwind of the success of Large Language Models (LLM). A family of usable large language models of codes in the open-source domain is Salesforce’s CodeGen. There has been a lot of success using LLMs and it seems as if the effectiveness is out of question.

What the question is here

On the other hand, Reinforcement Learning (RL) is a very brittle process with large variance throughout the training. Good general methods are rare and most of them still need domain-specific refinements for practical performances. Code generation as sequence modelling has been great. But how about the RL component? In other words:

Can RL actually solve sequential code-generating environments with execution results as feedbacks? How do methods compare (on-policy/off-policy, search-based, etc)? How do we do task-specific optimizations? …

What I planned to slowly try from here is to “scale-down”.

Let us completely ignore the language models for a moment and focus on what RL should do to play well with the situation. The current post marks the baby step of creating such a scaled-down environment for RL.

Here is the link.

What the game is

The rule of the game is simple. You start with an empty string "" and a target number (float, hidden by default). You have 17 buttons to push

  • 0 - 15 correspond to the characters 01234567890()+-*/
  • and a backspace as the extra action.

Reward

Every keystroke gets a feedback in the form of a reward. The reward is based on code execution and designed in the following way:

Notations:

  • let $acc$ be the accumulated reward of the episode;
  • let $x$ be the target number;
  • let $s$ be the string formed by the keystrokes until the current time step;
  • let $\text{ev}(s)$ be the evaluation result of the string $s$ as an arithmetic expression.

If the string $s$ is a valid arithmetic expression, we define the reward according to the following. $$ acc = \text{min}(100, \dfrac{1}{|\text{ev}(s) - x|})$$ If the string is invalid, $acc$ remains the same as the previous step. The reward at each step is the number that maintained $acc$ in the way described above.

Thoughts and Plans

The actions and rewards can be further tweaked for comparison. A couple ideas are:

  • removing the backspace and stop the environment once the expression fills up;
  • the current reward is very sparse. We could try reward engineering and create some partial rewards and punishments.

The plan is to implement and try various things on this environment. Interesting results will be posted later. I expect MCTS-based methods to be the strongest in this environment, and we will get to AlphaZero and MuZero at some point.
